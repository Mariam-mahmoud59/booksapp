## Architecture Overview

The application follows an **Offline-First Architecture**:

- **Local Database (SQLite):** The primary source of truth for all UI data.
- **Sync Queue:** All user actions are stored locally and queued.
- **Background Sync Engine:** Responsible for pushing local changes and pulling remote updates.
- **Supabase Backend:** Acts only as a remote persistence layer.

**Flow:**
`User Action в†’ Local DB в†’ Sync Queue в†’ Background Sync в†’ Supabase`

## Data Layer & Services

*   **`database_helper.dart`:** Handles all local database operations including CRUD and schema management.
*   **`supabase_service.dart`:** Handles communication with Supabase including authentication and remote data sync.
*   **`sync_service.dart`:** Implements FIFO queue-based synchronization using push-then-pull strategy.
*   **`connectivity_service.dart`:** Detects network changes and triggers synchronization when connection is restored.

## AI Usage in Development

AI tools are allowed ONLY as development assistants and must be used carefully.

**Rules:**
- AI can be used to generate code suggestions, fix bugs, or explain logic.
- All AI-generated code MUST be reviewed manually before use.
- Code must strictly follow the project architecture and file structure.
- Developers must fully understand any code before adding it.
- Blind copying from AI is strictly prohibited.

**Usage Guidelines:**
- **Use AI for:**
  - Debugging errors
  - Writing boilerplate code
  - Explaining complex logic
- **Do NOT use AI for:**
  - Changing project architecture
  - Modifying restricted files
  - Writing full features without understanding

**Validation Rule:**
Any code generated by AI must pass:
1. Architecture check
2. Code review
3. Testing

*Failure to follow these rules may lead to unstable code or project failure.*
