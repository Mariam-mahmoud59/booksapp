## Architecture Overview

The application follows an **Offline-First Architecture** with **Provider State Management**:

- **Local Database (SQLite):** The primary source of truth for all UI data.
- **Sync Queue:** All user actions are stored locally and queued.
- **Background Sync Engine:** Responsible for pushing local changes and pulling remote updates.
- **Supabase Backend:** Acts only as a remote persistence layer.
- **Provider (ChangeNotifier):** All screens consume data reactively through two providers.

### Data Flow
```
User Action → Provider → Repository → Local DB → Sync Queue → Background Sync → Supabase
```

### State Management

The app uses the `provider` package with two `ChangeNotifier` providers:

| Provider | File | Responsibility |
|----------|------|---------------|
| `StoryProvider` | `lib/providers/story_provider.dart` | Stories list, favorites, search, CRUD, pages, sync |
| `AuthProvider` | `lib/providers/auth_provider.dart` | Authentication, user profile, user stats |

Both providers are registered via `MultiProvider` in `main.dart`.

**Screen Usage Pattern:**
- Use `context.watch<StoryProvider>()` to **reactively rebuild** when data changes
- Use `context.read<StoryProvider>()` to **trigger actions** without rebuilding
- Same pattern applies to `AuthProvider`


## Data Layer & Services

| File | Responsibility |
|------|---------------|
| `database_helper.dart` | Handles all local SQLite operations including CRUD and schema management |
| `supabase_service.dart` | Handles communication with Supabase including authentication and remote data sync |
| `sync_service.dart` | Implements FIFO queue-based synchronization using push-then-pull strategy |
| `connectivity_service.dart` | Detects network changes and triggers synchronization when connection is restored |
| `story_repository.dart` | Single interface between providers and the data layer (DB + sync) |

## AI Usage in Development

AI tools are allowed **ONLY** as development assistants and must be used carefully.

### Rules
- AI can be used to generate code suggestions, fix bugs, or explain logic.
- All AI-generated code **MUST** be reviewed manually before use.
- Code must strictly follow the project architecture and file structure.
- Developers must fully understand any code before adding it.
- Blind copying from AI is **strictly prohibited**.

### Usage Guidelines
- **Use AI for:**
  - Debugging errors
  - Writing boilerplate code
  - Explaining complex logic
- **Do NOT use AI for:**
  - Changing project architecture
  - Modifying restricted files
  - Writing full features without understanding

### Validation Rule
Any code generated by AI must pass:
1. Architecture check
2. Code review
3. Testing

## General Rules

### 1. Core Rules
- Do **NOT** modify core project settings
- Do **NOT** change project structure
- Only edit files related to your assigned work
- Any unexpected change must be reported first
- Follow Provider state management strictly
- No pushing code without approval

### 2. AI Usage Policy
- AI is for assistance only
- Always review AI code manually
- Ensure code matches project architecture
- Never copy code blindly

### 3. Restricted Files (Do NOT Modify)
- `main.dart`
- `router.dart`
- Project structure
- Provider core logic (unless assigned)
- Theme files (unless you are Member 4)

### 4. Provider Rules
- All screens must consume data through `StoryProvider` or `AuthProvider`
- **Never** instantiate `StoryRepository()` or `SupabaseService()` directly in screens
- Use `context.watch<T>()` for reactive rebuilds, `context.read<T>()` for actions
- Do not add new providers without team approval

### 5. Development Workflow
1. Understand the task
2. Plan before coding
3. Implement changes
4. Test locally
5. Show results before pushing
6. Get approval
7. Push changes

### 6. Reporting Rules
- Every change must be documented
- Include what was changed and why
- Report any conflicts
- Do not modify other modules without coordination

### 7. Code Review Rules
- Code must be clean and readable
- Follow project structure
- No unnecessary changes
- Ensure no errors or warnings
- Match Provider pattern

### 8. Collaboration Rules
- Do not interfere with other modules
- Communicate before making shared changes
- Keep code consistent with team standards

### 9. Final Note
This document is mandatory. Any violation may cause conflicts or project failure.
